81 # We avoid calling the superclass constructor. This makes us about
82 # twice as fast (1.16 vs 0.68usec on PyPy, 29.3 vs 17.7usec on
83 # CPython 2.7), and has the important effect of avoiding
84 # allocation and then immediate deletion of _Thread__block, a
85 # lock. This is especially important on PyPy where locks go
86 # through the cpyext API and Cython, which is known to be slow and
87 # potentially buggy (e.g.,
88 # https://bitbucket.org/pypy/pypy/issues/2149/memory-leak-for-python-subclass-of-cpyext#comment-22347393)
89  
90 # These objects are constructed quite frequently in some cases, so
91 # the optimization matters: for example, in gunicorn, which uses
92 # pywsgi.WSGIServer, most every request is handled in a new greenlet,
93 # and every request uses a logging.Logger to write the access log,
94 # and every call to a log method captures the current thread (by
95 # default).
96 #
97 # (Obviously we have to duplicate the effects of the constructor,
98 # at least for external state purposes, which is potentially
99 # slightly fragile.)
100  
101 # For the same reason, instances of this class will cleanup their own entry
102 # in ``threading._active``
103  
104 # This class also solves a problem forking process with subprocess: after forking,
105 # Thread.__stop is called, which throws an exception when __block doesn't
106 # exist.
